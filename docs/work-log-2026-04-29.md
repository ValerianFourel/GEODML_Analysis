## 1. Pre/post-run pipeline audit (commit `16af71e`)

**Why:** the port spans Stages A→D plus Stage F (interp). Without a single
audit point, it was impossible to tell at a glance how a chain was progressing
or what changed between a "before" and "after" snapshot of a run.

**What:** new `scripts/audit_pipeline.py` reports, per variant:

- **Inputs**: cached SERP parquets + HTML tarballs (warns if missing — the
  most common reason rerank silently produces 0 keywords).
- **Stage A**: per `(model, engine, pool, variant)` keyword counts in each
  `keywords.jsonl`, plus run-dir size.
- **Stage B**: per-`(engine, pool)` features parquet rows (variant-agnostic).
- **Stage C**: merged main parquet rows + cols.
- **Stage D**: DML parquet rows + cross-variant headline (POOLED + plr +
  lgbm + rank_delta) showing biased vs neutral coefficients side-by-side
  with Δ.
- **Live `squeue` summary** in the header (R/PD/CG counts).

Two modes:

```bash
python scripts/audit_pipeline.py --save audits/before.json
# ...run jobs...
python scripts/audit_pipeline.py --save audits/after.json
python scripts/audit_pipeline.py --compare audits/before.json audits/after.json
```

Verified with synthetic snapshots: compare-mode correctly highlights only the
deltas. JSON snapshot format includes everything in the live report.

## 3. Order-sensitivity probe (commit `a3dbf00`)

**Why:** every DML coefficient downstream is conditional on the LLM doing
real ranking, not just regurgitating SERP order. Without an explicit test, we
cannot tell whether a +0.8 rank-delta on T7 reflects judgment or anchoring.

**Design (operator-confirmed):**

- Two random shuffles per cell with seeds 42 and 123 (lets us also report
  shuffle-vs-shuffle overlap as a noise floor).
- Cover both prompt variants and the full keyword set.
- `temperature=0.1` to match the main pipeline.
- Total: 32 jobs ≈ 32k LLM calls ≈ ~64 GPU-hours.

**What landed:**

| Path | Purpose |
|---|---|
| `interpretability/pipeline/order_probe.py` | Reuses `rank_one_keyword()`. Per-keyword RNG `random.Random(f"{seed}::{keyword}")` so adding new keywords is stable. Output JSONL has `seed`, `input_order_perm`, `original_order` extra fields. `--smoke` flag uses a CPU mock ranker for unit testing. |
| `interpretability/pipeline/order_probe_analyze.py` | Pairwise Jaccard / overlap@K (K∈{3,5,10}) → `data/order_probe/order_probe_summary.parquet`. Inner-join on keywords across all three orderings. |
| `scripts/slurm/run_order_probe.sbatch` | Booster, 4 GPU, 23:55 walltime, chain-resubmit. |
| `scripts/slurm/dispatch_all.sh` | New `--only order_probe` (+ `--seeds`, `--order-probe-variants` flags). |
| `scripts/audit_pipeline.py` | New Stage A' section + headline analysis summary. |
| `interpretability/make_figures.py` | `figure_e_order_sensitivity`: per-cell boxplots + paired Δ(neutral − biased) with bootstrap CI. |
| `docs/order-probe.md` | Operator runbook + interpretation thresholds. |

**Tested locally:**

- Shuffle is deterministic per `(seed, keyword)`; stable when keyword set
  changes; produces a valid permutation.
- End-to-end smoke via `--smoke` writes the correct envelope, prompt is
  rendered from the shuffled list, `--resume` is idempotent.
- Analyzer recovers hand-rigged overlaps to 1e-9 (e.g. set-orig and set-42
  share 7/10 → Jaccard 7/13, overlap@10 = 0.7).

## 4. Stage F variant-awareness (commit `abe0cf1`)

**Why:** `interpretability.{ablation,saliency,probing,weight_analysis}` wrote
to flat un-suffixed dirs (`ablation_T7_Llama/`, `saliency_Llama/`, …). Running
the same model under both variants would clobber the first variant's outputs.

**Approach:** keep the Python modules unchanged (they accept `--output-dir`
already); push variant routing into the sbatch wrappers and add a one-time
migration helper for legacy outputs.

**What landed:**

- All four sbatch scripts (`run_ablation`, `run_saliency`, `run_probing`,
  `run_weights`) now read `PROMPT_VARIANT` and embed it in `OUTDIR`:
  `interpretability/output/<stage>_..._<variant>/`. `chain_resubmit` forwards
  the variant. `run_probing` also picks the variant-suffixed run dir for
  `RUN_FILTER` so probing reads the correct rerank output.
- `dispatch_all.sh` was already passing `PROMPT_VARIANT` for ablation /
  saliency / probing — fixed `weights` to do the same.
- `scripts/slurm/merge_ablation.sh` rewritten variant-aware: writes
  `ablation_results_{full,rw}_<variant>.csv` and a legacy alias when
  variant=biased.
- `scripts/slurm/merge_interp.sh` (new) does the equivalent for saliency /
  probing / weights — produces top-level merged CSVs that the figures read.
- `scripts/migrate_interp_outputs_to_variant.sh` (new) renames pre-port
  un-suffixed dirs to `_biased`. Idempotent, dry-run by default, applied
  on JUWELS — moved 18 directories (12 ablation + 2 saliency + 2 probing +
  2 weights).
- `scripts/audit_status.py`: `--variant biased|neutral|both` (default both);
  iterates per-variant cells with the new dir naming.
- `scripts/audit_pipeline.py`: new Stage F section in live report and in
  snapshots; compare-mode reports interp deltas per `(stage, model, variant,
  treatment|frame)`.
- `interpretability/make_figures.py`: `figure_a/b/c/d` auto-detect variant-
  suffixed merged CSVs and emit per-variant PNGs plus biased legacy aliases.
- `docs/pipeline-port.md` §5 updated with the new flow + migration step.

**Tested locally** with synthetic CSVs: `merge_ablation` and `merge_interp`
produce correct rows and legacy aliases; `figure_a/b/c/d` emit per-variant
plots end-to-end (`figure_a_ablation_full_biased.png`,
`figure_a_ablation_full_neutral.png`, plus legacy `figure_a_ablation_full.png`).
All sbatch scripts pass `bash -n`. dispatch_all.sh dry-run shows all stages
now embed the variant in job name + exports.
