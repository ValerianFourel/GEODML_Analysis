## Phase 4 — First passage attempt + critical bug discovered

Smoke test (`MAX_KW=20`) ran, hit the API ~480 times, all "succeeded" — but every cell logged:

```
[rerank] passages: total=9,141 missing_html=9,141 empty_extract=0
```

**100% of URLs failed HTMLLoader lookup.** The smoke "completed" but every passage in every prompt was empty.

Diagnostic:
```python
# new pipeline:
hashlib.md5(url.encode("utf-8")).hexdigest()[:16] + ".html"   # 0/200 hits

# upstream paperSizeExperiment (gather_data.py, extract_features.py, ...):
hashlib.sha256(url.encode()).hexdigest()[:16]                  # 4898/5296 hits = 92.5%
```

The new pipeline's `url_to_html_filename` had used **MD5** since the repo's first commit, but every upstream cache writer uses **SHA-256**. Every passage variant in the project's entire history (cluster + smoke) was functionally a snippet variant — the LLM saw `passage:` followed by an empty string for every result. Verified by reading actual cluster prompts:

```
1. [help.shopify.com] Recovering abandoned checkouts
   snippet: Items in a customer's cart...
   passage:                              ← empty, all the way down
```

Worse: even with the SHA-256 fix, only ~25% of phase0 SERP URLs match the cache because phase0 and the original scrape are from different DataForSEO billing batches with disjoint URL sets.

## Phase 5 — Fix and verify

`interpretability/utils.py`:
```python
def url_to_html_filename(url: str) -> str:
    """Convention used by the upstream paperSizeExperiment pipeline:
    sha256(url)[:16] + '.html'.

    Must match exactly what the scraper wrote — see
    GEODML/pipeline/gather_data.py:_url_to_cache_key (and identical
    definitions in extract_features.py, rebuild_features.py,
    fill_confounders.py, save_phase2_progress.py, export_html_cache.py,
    earned_media_halo.py).
    """
    return hashlib.sha256(url.encode()).hexdigest()[:16] + ".html"
```

### Caveat about T5_topical_comp

The original GEODML paper headlined `T5_topical_comp = +0.438 (strong promoter)`. Our run reproduces this only for RF (+0.40, p=0.07); LGBM gives +0.23 (n.s.). Adding DataForSEO competitiveness confounders (which weren't in the original headline grid) absorbs about half the apparent T5 effect. This is itself a methodological refinement worth noting in the paper.

## Bugs found today

1. **MD5 vs SHA-256 in `url_to_html_filename`** — the big one. `interpretability/utils.py` had used MD5 since the repo's first commit; upstream cache writers use SHA-256. Every passage variant ever run with this code had empty passages. Fix: 1-line change in `utils.py` + wipe and re-run.

2. **`scripts/build_main_table.py` ModuleNotFoundError** — script imported `interpretability.*` but didn't add the repo root to `sys.path`. Fixed with prepend in the script header.

3. **Stage B torch/numpy mismatch** — `torch==2.2.2` and `numpy==2.4.4` in the venv, but recent `transformers` requires `torch>=2.4` and `numpy<2`. Worked around by using `--no-embed` (skipping the sentence-transformer); permanent fix is `pip install --upgrade torch>=2.4 numpy<2`.

4. **DML duplicate-process race** — at one point two `python -m interpretability.pipeline.dml --variant neutral` ran concurrently (one from the background task I started, one from the user's `finish_via_api.sh` chain). Both used `--resume`, so no data corruption, but they did duplicate work and slowed each other down. Lesson: don't background-spawn long pipeline runs while the user is also running the same orchestrator.

5. **DML "binary predictions" error on small-data passage cells** — DoubleML's sanity check refuses to use propensity-model predictions that look like binary labels (0/1) instead of probabilities (fractional). Caused by LightGBM overfitting on the smoke 1,400-row passage data. Real fix is more data (the full passage rerun); not a code bug.

6. **`build_dataset_mirror.sh` first attempt ran out of disk** at 6.9 GB used. User freed disk space, second attempt with `FORCE=1` completed.

7. **`finish_via_api.sh` `MAX_KW=20` triggered passage extraction over ALL serp URLs** — the keyword cap only limits LLM calls, not passage extraction. With the disk passage cache I added later (`5eb9eb5`), the first cell pays the extraction cost and all subsequent cells load from disk in milliseconds.
