## 1. Order-probe per-cell report (commit `e9df432`)

**Why:** the headline `order_probe_summary.parquet` had 96k rows but the only
view we had on it was the K=10 mean by `(variant, ordering_pair)` printed at
the end of the analyzer — three rows per variant. That was enough to see the
biased−neutral gap (≈0.69 vs ≈0.82 mean overlap@10) but not enough to know
*where* the gap concentrated.

**What landed:** `interpretability/pipeline/order_probe_report.py` slices the
summary into four operator-friendly CSVs in `interpretability/output/`:

- `order_probe_by_cell.csv` — per `(variant, model, engine, pool)` at K=10:
  `mean_jacc`, `mean_oak`, `p10_oak` (10th-percentile keyword tail), counts.
- `order_probe_biased_minus_neutral.csv` — per `(model, engine, pool)`,
  `Δoak = biased − neutral`, sorted ascending so the cells where biased is
  *most* order-sensitive vs neutral float to the top.
- `order_probe_k_trend.csv` — `mean_jacc` / `mean_oak` per `(variant, K)` for
  K ∈ {3, 5, 10}.
- `order_probe_worst_keywords.csv` — 30 keywords with the lowest mean overlap.

**What we learned from the slices** (these are the empirical hooks for the
paper, written down here so they don't get lost):

1. **The Δoak gap is concentrated in `searxng/pool=50`** (-0.27 for both
   models). DDG cells show much smaller gaps (-0.05 to -0.12). The
   prompt-induced position-bias amplification headline is mostly a
   `searxng/pool=50` phenomenon.
2. **K=10 is not monotone in K for biased.** Biased mean_oak: 0.640→0.704→0.687
   (peaks at K=5, dips at K=10). Neutral monotone increases:
   0.606→0.660→0.855. Reads as "biased = a few star items + noise; neutral =
   stable winner set, just permuted within."
3. **The biased-prompt's exclusion list is doing damage.** Many of the 30
   worst-overlap keywords have `mean_oak ≈ 0.10` and `mean_jacc = 1.00` — i.e.
   the LLM returned only 1 valid domain on both runs, and it was the same
   domain. This happens for queries like *"stress relief techniques"*,
   *"how to manage stress"*, *"anxiety disorder treatment options"* where the
   biased prompt's "software product" instruction has no eligible candidates.
   None of these show up under neutral.

## 2. Passage-augmented (RAG) rerank arm (commit `b1b1c51`)

**Motivation.** External-validity research showed every production
generative-search system (Perplexity Sonar, Google AI Overviews, Bing Copilot,
ChatGPT Search) and the original GEO benchmark (Aggarwal et al., KDD '24) feeds
the LLM **passage-level body text**, not SERP snippets. The 150-char snippet
condition is the thinnest information unit any of these regimes uses. If
findings (e.g. position-bias amplification under biased prompts) only hold
under that bottleneck, they may not generalize to real GEO. Bracketing
experiment: run the same 8 cells × 2 prompt variants again, but feed the LLM
~800 chars of cleaned body text per result alongside the snippet.

### Design — 4 variants, not 2

Extended the existing variant axis from 2 to 4:

```
biased            neutral            ← snippet-only, ~150 chars (existing)
biased_passage    neutral_passage    ← +800 chars body text (new)
```

Rationale: variant is already a string suffix on the run_id (`config.py:186`
`run_label_with_variant`); every downstream consumer accepts a variant list.
Reusing the suffix mechanism kept the change to ~15 small edits across
plumbing files plus one new helper. Adding a separate orthogonal "content
mode" axis would have required signature changes in many more places.

### Per-result rendering

Snippet variants (unchanged, byte-identical to today — verified at runtime):

```
{position}. [{domain}] {title} — {snippet[:150]}
```

Passage variants emit a multi-line block:

```
{position}. [{domain}] {title}
   snippet: {snippet[:150]}
   passage: {passage[:800]}
```

Span semantics matter for downstream interpretability:

- `domain_span` continues to point to the bare domain inside `[...]`.
  Saliency T7 binding (`saliency.py:308`) stays correct.
- `line_span` now covers the **full multi-line block** for passage variants —
  probing's per-result hidden-state pool (`probing.py:127–135`) therefore
  spans the body content too, which is the experimental signal.

### Passage extraction

`interpretability/utils.py:extract_passage(html, max_chars=800)`:

1. Calls `trafilatura.extract(html, include_comments=False, include_tables=False, favor_recall=True)` for boilerplate-stripped article text.
2. Falls back to existing `page_digest()` when trafilatura yields `None` or
   `<100 chars` (covers JS-heavy pages and weird markup).
3. Strips `[`, `]`, and newlines so the result interpolates safely into the
   bracketed multi-line block.
4. Truncates to `max_chars`.

Lazy-imports trafilatura so machines without it can still import the module
(falls through to `page_digest`).

### HTML source path — critical detail

The html_cache lives at the **un-suffixed cell-level path**
`data/runs/{engine}_{model}_serp{N}_top{K}/phase2/html_cache/`, not the
variant-suffixed path. Smoking-gun is `features.py:665` which already skips
any `*_biased` / `*_neutral` directory when looking for HTML. So
`_build_passage_map()` opens an `HTMLLoader` keyed on `C.run_label(...)` (no
variant suffix); the passage cache is shared across both prompt variants of a
cell. Single one-shot extraction per cell, ~5K-13K unique URLs, ~30-60 s
walltime via trafilatura.

### Token budget

Per-result block in passage mode: ~150 (snippet) + ~800 (passage) + ~80
(boilerplate) ≈ 1030 chars / ~260 tokens. For pool=50: ~13K input tokens.
Both Llama-3.3-70B and Qwen2.5-72B have ≥128K context. `LLM_MAX_TOKENS=500`
(`config.py:53`) is the *response* cap, irrelevant.

### Headline diagnostic comparisons

`scripts/compare_variants.py --pair-grid` will emit, per cell, the four cells
of the bracketing experiment:

1. `..._biased` vs `..._biased_passage` — passage effect within biased.
2. `..._neutral` vs `..._neutral_passage` — passage effect within neutral.
3. `..._biased` vs `..._neutral` — existing prompt-instruction effect (sanity).
4. `..._biased_passage` vs `..._neutral_passage` — does body content
   neutralize the prompt-instruction gap?

**Reading rule for the paper.** If `biased_passage ≈ neutral_passage` while
`biased ≠ neutral`, the prompt-instruction effect is a *snippet artifact* and
doesn't generalize to production GEO — that's the more interesting paper. If
the gap persists across both information conditions, prompt-instruction bias
is robust to information richness, which is also publishable.

## 3. Cluster-ops incidents (no code changes — but worth not re-discovering)

### 3.1 `pip install --user trafilatura` blew up `$HOME` quota

`pip install --user` writes to `~/.local/`, and JUWELS home directories are
small (~10–20 GB). 6 of 14 packages installed before `OSError: Disk quota
exceeded`.

**Fix:** install into the venv directly (no `--user`). The venv at
`/p/project1/obdifflearn/vfourel/venvs/geodml/` lives on a project filesystem
with real space.

### 3.2 The venv had no pip

`pip install trafilatura` (no `--user`) succeeded — but installed into
`/p/scratch/.../miniforge3/lib/python3.12/site-packages/`, the **miniforge3
base**, not the venv. `which python` was `/p/project1/.../venvs/geodml/...`,
`which pip` was `/p/scratch/.../miniforge3/bin/pip`. Mismatch. Then
`python -m pip install` failed with `No module named pip` because the venv
was created `--without-pip` (intentional pattern on JSC HPC envs).

**Fix sequence:**

```bash
python -m ensurepip --upgrade --default-pip   # bootstraps pip into the venv
python -m pip --version                        # now resolves to the venv's pip
python -m pip install trafilatura
python -c "import trafilatura; print(trafilatura.__version__)"
```

### 3.3 `--partition=batch` is hardcoded; account is Booster-only

`scripts/slurm/run_pipeline.sbatch:21` and `scripts/slurm/run_dml.sbatch:18`
are pinned to `#SBATCH --partition=batch` (JUWELS Cluster). The
`$JUWELS_ACCOUNT` budget for this project is Booster-only, so submission via
the orchestrator returned

```
sbatch: error: Batch job submission failed: Invalid account or account/partition combination specified
```

**Workaround pattern, applied today:**

- For `dml`: `./scripts/slurm/dispatch_all.sh --only dml --variant X --partition booster --time 05:55:00`.
- For the orchestrator (`run_pipeline.sbatch`): skip it entirely. Use
  per-stage `dispatch_all.sh --only {rerank,features,ablation,saliency,probing,weights,order_probe}`
  — those underlying sbatch files (`run_rerank.sbatch`, `run_ablation.sbatch`,
  …) all default to `--partition=booster`, no override needed.

Long-term fix would be making the `batch` partition configurable via env in
both sbatch files; deferred — the workaround is a one-line override.

### 3.4 `$JUWELS_ACCOUNT` is not sticky across login sessions

Hit twice today. `.env` has the account, but each fresh shell forgets it.
Diagnosis: `echo "$JUWELS_ACCOUNT"` prints empty → sbatch sees `--account=`
(literally empty) → "Invalid account" error.

**Fix:**

```bash
set -o allexport; source .env; set +o allexport
```

Recommended addition to `~/.bashrc` (planned, not yet committed):

```bash
[ -f "$HOME/Hamburg/GEODML_Analysis/.env" ] && {
  set -o allexport; . "$HOME/Hamburg/GEODML_Analysis/.env"; set +o allexport;

```

### 3.5 Trailing flags after the script path don't override sbatch

User attempted `sbatch ... run_pipeline.sbatch --partition booster`. SLURM
treats trailing args as `$1`, `$2` to the script — they don't reach sbatch's
own argument parser. All `--partition`, `--time`, `--account` overrides must
go *before* the script path.

### 6.5 Reading the result

The headline number is mean Kendall's τ in
`compare_variants.py`'s output for `..._biased` vs `..._biased_passage`
across cells. If τ stays ≥ 0.7, the snippet-only finding is robust to
passage augmentation and the paper can claim generalization to production
GEO. If τ collapses (≤ 0.4), the passage condition is doing meaningful work
and the paper has the sharper "snippet-only listwise rerankers don't
generalize" story instead.

The four pair-delta CSVs from `order_probe_report.py` are the secondary
diagnostics — particularly `order_probe_biased_passage_minus_neutral_passage.csv`,
which tells us whether the prompt-instruction gap survives once the LLM has
body text to anchor on.
